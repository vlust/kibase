│ Plan to implement                                                            │
│                                                                              │
│ kibase: KiCad Project Template Plan                                          │
│                                                                              │
│ Context                                                                      │
│                                                                              │
│ The goal is to build kibase — a GitLab-hosted monorepo template for managing │
│  KiCad PCB projects with structured collaborative workflows, automated asset │
│  generation (via KiBot + Docker), and per-project semantic versioning. The   │
│ repo will serve as the foundation for multiple independent boards in a       │
│ single git repo, with shared libraries via submodule and a clear Draft →     │
│ Review → Released design stage system.                                       │
│                                                                              │
│ Two reference projects exist in the repo and were analyzed:                  │
│ - KDT_Hierarchical_KiBot/ — comprehensive GitHub Actions + KiBot template    │
│ (4-stage, many YAML files)                                                   │
│ - kicad-ci-cd-pipeline/ — simpler GitLab CI + semantic-release example       │
│                                                                              │
│ This template draws from both, adapting for GitLab, 3-stage workflow,        │
│ per-project versioning, and JLCPCB support.                                  │
│                                                                              │
│ ---                                                                          │
│ Key Decisions                                                                │
│                                                                              │
│ ┌───────────────┬─────────────────────────────────────────────────────────┐  │
│ │    Concern    │                        Decision                         │  │
│ ├───────────────┼─────────────────────────────────────────────────────────┤  │
│ │ Repo layout   │ Monorepo: projects/, libraries/ (submodule), kibot/,    │  │
│ │               │ root CI                                                 │  │
│ ├───────────────┼─────────────────────────────────────────────────────────┤  │
│ │ Design stages │ Draft → Review → Released                               │  │
│ ├───────────────┼─────────────────────────────────────────────────────────┤  │
│ │ Committed     │ PDFs (schematic + layout) committed; fab files          │  │
│ │ assets        │ (gerbers, BOM, positions) as CI artifacts only          │  │
│ ├───────────────┼─────────────────────────────────────────────────────────┤  │
│ │ Local tooling │ Docker + KiBot (same image as CI)                       │  │
│ ├───────────────┼─────────────────────────────────────────────────────────┤  │
│ │ Versioning    │ Per-project, triggered by merge to main                 │  │
│ ├───────────────┼─────────────────────────────────────────────────────────┤  │
│ │ Commit        │ change(proj): → minor bump; redesign(proj): → major     │  │
│ │ prefixes      │ bump; others → no bump                                  │  │
│ ├───────────────┼─────────────────────────────────────────────────────────┤  │
│ │ Libraries     │ Git submodule at libraries/                             │  │
│ ├───────────────┼─────────────────────────────────────────────────────────┤  │
│ │ JLCPCB        │ KiBot native JLCPCB output; LCSC field in components    │  │
│ └───────────────┴─────────────────────────────────────────────────────────┘  │
│                                                                              │

│ ---                                                                          │
│ Phase 2: KiBot Configuration                                                 │
│                                                                              │
│ Three-stage KiBot setup. Base the structure on the KDT reference             │
│ (KDT_Hierarchical_KiBot/kibot_yaml/) but simplified to ~4 files instead of   │
│ 38.                                                                          │
│                                                                              │
│ kibot/globals.yaml                                                           │
│                                                                              │
│ - Project metadata variables (company, project name, revision)               │
│ - JLCPCB drill/gerber format settings                                        │
│ - Color theme reference                                                      │
│                                                                              │
│ kibot/filters.yaml                                                           │
│                                                                              │
│ - Filter: exclude DNP components from BOM/positions                          │
│ - Filter: rename fields for JLCPCB BOM format (LCSC → LCSC Part #)           │
│ - Filter: exclude testpoints from placement                                  │
│                                                                              │
│ kibot/draft.yaml                                                             │
│                                                                              │
│ - Output: schematic PDF (with "DRAFT" text variable injected)                │
│ - No ERC/DRC, no PCB outputs                                                 │
│ - Fast — for local WIP sharing                                               │
│                                                                              │
│ kibot/review.yaml                                                            │
│                                                                              │
│ Inherits globals + filters. Outputs:                                         │
│ - Schematic PDF (multi-sheet, A4)                                            │
│ - PCB layout PDF (top + bottom assembly views)                               │
│ - ERC report (run but non-blocking)                                          │
│ - DRC report (run but non-blocking)                                          │
│ - Interactive HTML BOM (for MR review convenience)                           │
│                                                                              │
│ kibot/release.yaml                                                           │
│                                                                              │
│ Inherits review outputs plus:                                                │
│ - Gerbers (KiCad 6+ format, JLCPCB layer names)                              │
│ - Drill files (Excellon, JLCPCB format)                                      │
│ - BOM CSV (JLCPCB format: Designator, Comment, Footprint, LCSC Part #)       │
│ - Position/CPL file (with JLCPCB rotation offsets applied via KiBot's        │
│ built-in correction)                                                         │
│ - All outputs go to output/ directory (not committed)                        │
│                                                                              │
│ ---                                                                          │
│ Phase 3: Local Generation Script                                             │
│                                                                              │
│ scripts/generate.sh                                                          │
│                                                                              │
│ Usage: ./scripts/generate.sh <project-path> <stage> [options]                │
│   project-path:  e.g. projects/example                                       │
│   stage:         draft | review | release                                    │
│                                                                              │
│ - Pulls/uses ghcr.io/inti-cern/kibot:dev (or pinned version)                 │
│ - Mounts repo root into container                                            │
│ - Runs KiBot with the appropriate kibot/<stage>.yaml                         │
│ - Outputs land in <project-path>/output/ locally                             │
│                                                                              │
│ ---                                                                          │
│ Phase 4: GitLab CI/CD Pipeline                                               │
│                                                                              │
│ .gitlab-ci.yml Stages                                                        │
│                                                                              │
│ validate (all branches, on MR)                                               │
│ - Detect changed projects via scripts/detect-changed-projects.sh             │
│ - Run ERC + DRC on each changed project                                      │
│ - Fail MR if critical ERC errors found (configurable threshold)              │
│                                                                              │
│ generate (main branch + MR branches)                                         │
│ - For each changed project: run review stage KiBot                           │
│ - On main: also commit updated PDFs back to projects/<name>/docs/            │
│ - On MR: expose PDFs as downloadable CI artifacts                            │
│                                                                              │
│ release (main branch only, after generate)                                   │
│ - Run scripts/version-bump.py to detect change(proj): / redesign(proj):      │
│ commits since last tag                                                       │
│ - For each bumped project:                                                   │
│   - Update projects/<name>/CHANGELOG.md                                      │
│   - Create git tag <project-name>/v<version>                                 │
│   - Run release stage KiBot                                                  │
│   - Create GitLab release with fab file artifacts attached                   │
│   - Artifacts: gerbers zip, BOM CSV, position CSV                            │
│                                                                              │
│ scripts/detect-changed-projects.sh                                           │
│                                                                              │
│ git diff --name-only $CI_COMMIT_BEFORE_SHA $CI_COMMIT_SHA \                  │
│   | grep '^projects/' | cut -d/ -f2 | sort -u                                │
│                                                                              │
│ ---                                                                          │
│ Phase 5: Per-Project Versioning Script                                       │
│                                                                              │
│ scripts/version-bump.py                                                      │
│                                                                              │
│ Since semantic-release doesn't natively support path-scoped monorepo         │
│ versioning, a lightweight Python script handles this:                        │
│                                                                              │
│ 1. For each project name passed in:                                          │
│   - Find the last git tag matching <project>/v*                              │
│   - Get all commits since that tag touching projects/<project>/              │
│   - Parse commit messages for change(<project>): (minor) or                  │
│ redesign(<project>): (major)                                                 │
│   - Determine new version (semver)                                           │
│   - Prepend new version section to projects/<project>/CHANGELOG.md using     │
│ commit messages                                                              │
│   - Write version to projects/<project>/VERSION file (also used by KiBot via │
│  text variable)                                                              │
│   - Create annotated git tag <project>/v<version>                            │
│                                                                              │
│ Commit message format (enforced by convention, not tooling):                 │
│ change(board-a): add decoupling caps to 3V3 rail                             │
│ redesign(board-a): switch from STM32F4 to RP2040                             │
│ chore: update CI image version           ← no bump                           │
│ docs: update README                      ← no bump                           │
│                                                                              │

│ ---                                                                          │
│ Phase 7: KiRI Visual Diff Integration                                        │
│                                                                              │
│ KiBot has a native kiri output plugin (confirmed in                          │
│ KDT_Hierarchical_KiBot/kibot_yaml/kibot_out_html_kiri.yaml). This generates  │
│ an HTML-based interactive diff viewer showing schematic and PCB changes      │
│ between commits.                                                             │
│                                                                              │
│ KiBot Config Addition                                                        │
│                                                                              │
│ Add to kibot/review.yaml:                                                    │
│ - name: kiri_diff                                                            │
│   type: kiri                                                                 │
│   dir: kiri                                                                  │
│   options:                                                                   │
│     max_commits: 0          # 0 = all commits since branch point             │
│     revision: '%KIRI_BASE%' # injected by CI as merge base SHA               │
│     zones: global                                                            │
│     keep_generated: true                                                     │
│     layers: all                                                              │
│                                                                              │
│ The revision is set to the MR's merge base ($CI_MERGE_REQUEST_DIFF_BASE_SHA) │
│  by CI, so KiRI shows exactly what changed in the MR — not just the last N   │
│ commits.                                                                     │
│                                                                              │
│ GitLab CI Integration                                                        │
│                                                                              │
│ GitLab Pages approach (chosen for best UX):                                  │
│                                                                              │
│ GitLab Pages serves a single public/ directory. To support per-MR URLs       │
│ without wiping previous deployments, a dedicated kiri-pages branch maintains │
│  the accumulated output:                                                     │
│                                                                              │
│ CI flow on MR push:                                                          │
│ 1. Generate KiRI output → output/kiri/                                       │
│ 2. Checkout kiri-pages branch (or create if missing)                         │
│ 3. Copy output to public/mr-{IID}/{project}/                                 │
│ 4. Commit and push kiri-pages branch                                         │
│ 5. GitLab Pages serves https://org.gitlab.io/repo/mr-42/board-a/             │
│                                                                              │
│ CI flow on MR merge/close:                                                   │
│ - Remove public/mr-{IID}/ from kiri-pages branch (cleanup)                   │
│                                                                              │
│ MR comment posted via GitLab API:                                            │
│ curl --request POST \                                                        │
│   --header "PRIVATE-TOKEN: $GITLAB_BOT_TOKEN" \                              │
│   --data "body=KiRI diff:                                                    │
│ https://${CI_PAGES_DOMAIN}/mr-${CI_MERGE_REQUEST_IID}/${PROJECT_NAME}/" \    │
│   "${CI_API_V4_URL}/projects/${CI_PROJECT_ID}/merge_requests/${CI_MERGE_REQU │
│ EST_IID}/notes"                                                              │
│                                                                              │
│ Fallback (simpler, no Pages setup required):                                 │
│ Use artifacts: expose_as to add a direct "KiRI Diff" link in the MR pipeline │
│  tab:                                                                        │
│ artifacts:                                                                   │
│   expose_as: 'KiRI Diff'                                                     │
│   paths:                                                                     │
│     - output/kiri/                                                           │
│   expire_in: 30 days                                                         │
│ This is a good starting point — implement Pages enhancement later.           │
│                                                                              │
│ New CI Job (kiri in GitLab CI)                                               │
│                                                                              │
│ kiri:                                                                        │
│   stage: review                                                              │
│   rules:                                                                     │
│     - if: $CI_PIPELINE_SOURCE == "merge_request_event"                       │
│   script:                                                                    │
│     - detect changed projects                                                │
│     - for each project: run KiBot with review.yaml,                          │
│ KIRI_BASE=$CI_MERGE_REQUEST_DIFF_BASE_SHA                                    │
│     - push KiRI output to kiri-pages branch                                  │
│     - post MR comment with link                                              │
│   artifacts:                                                                 │
│     expose_as: 'KiRI Diff'    # fallback: always works                       │
│     paths: [output/kiri/]                                                    │
│     expire_in: 30 days                                                       │
│                                                                              │
│ Prerequisite: CI Access Token                                                │
│                                                                              │
│ - A project access token or deploy key with write_repository scope is needed │
│  to push to the kiri-pages branch                                            │
│ - Store as GITLAB_BOT_TOKEN CI variable                                      │
│                                                                              │

❯ when does kiri diff get triggered?                                            

● KiRI diff runs in the review:kiri job, which triggers only on merge request
  events:

  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"

  So it fires whenever you push to a branch that has an open MR. It does not run
   on plain branch pushes without an MR, and does not run on main.

  The base commit it diffs against is $CI_MERGE_REQUEST_DIFF_BASE_SHA — the
  merge base between your branch and the target branch — so the diff always
  shows exactly what changed in the MR, regardless of how many commits are on
  the branch.

● You only need the prefix once — on the merge commit (or squash commit). Every
  individual commit on a feature branch can say whatever you want. The version
  script only scans commits that landed on main.

  So the workflow is actually:

  feature branch:   fix typo, wip, tweak values, another tweak   ← write
  anything
                                      ↓ merge MR
  main:             change(example): add decoupling caps           ← one
  intentional message

  GitLab prompts you for the merge/squash commit message — that's the only place
   the convention applies.

  ---
  That said, you're right to question whether even that is too much friction.
  Two alternatives worth considering:

  Option A — auto-bump on folder change (no convention needed)
  Any merge to main that touches projects/<name>/kicad/ auto-bumps the version.
  You'd lose the ability to signal minor vs major, but gain zero overhead.

  Option B — auto-patch, explicit for minor/major
  - Touch kicad/ → patch bump automatically
  - Write change(name): → minor bump
  - Write redesign(name): → major bump

  This is probably the best balance: small iterative work requires nothing, you
  only write a prefix when something is actually significant.

  My recommendation is Option B. It means you almost never have to think about
